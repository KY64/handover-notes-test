Goal  
Build a POC full-stack hotel shift handover application for Lumen Boutique Hotel. The app should help staff:

- View structured handover events.
- Identify pending/unresolved carry-over tasks.
- Add/update handover notes.
- Reconcile free-text night logs into structured events.
- Normalize multilingual notes into English.
- Protect the API with a configurable sliding-window rate limit.

Desired outcome / acceptance criteria  
A practical POC should probably be considered successful when:

1. Staff can open a web UI and see handover items sorted by urgency/status.
2. Existing `sample/events.json` can be loaded as seed data.
3. Free-text notes from `sample/night-logs.md` can be submitted and converted into structured event-like records.
4. Chinese/mixed-language notes are translated or normalized into English before storage/display.

Key questions / tradeoffs  

1. Should the POC use a real Postgres database immediately, or start with file/in-memory storage and migrate later?
   - Postgres is closer to production and Render deployment.
   - File/in-memory is faster, but less representative.

2. Should AI reconciliation call OpenRouter for real, or should we first build a deterministic/mock reconciliation path?
   - Real OpenRouter proves the core product value.
   - Mock mode avoids API key/deployment friction and makes tests stable.

3. Should free-text reconciliation automatically write events to the log, or create a review screen first?
   - Auto-write is simpler.
   - Review-before-import is safer, especially because hotel notes may contain ambiguity or prompt-injection-like guest text.

4. Should priority be a stored field, or computed dynamically from `type`, `status`, age, and keywords?
   - Stored priority is simple for staff.
   - Computed priority avoids stale priority and is better for a POC with changing rules.

5. Should the frontend be fully static SvelteKit calling a separate Hono API?
   - This matches `DESIGN.md`.
   - It means local dev/deployment needs two services.

6. What is the minimum authentication story?
   - For a POC, maybe no login, only rate limiting.

7. Should the app preserve the original raw free-text note alongside extracted structured events?

## Question 4: priority — stored field vs computed field

The question was whether priority should be stored directly in the database or computed dynamically from event data.

### Option A: Stored priority

Example:

```ts
priority: "critical" | "high" | "medium" | "low"
```

#### Pros

- Simple for frontend sorting/filtering.
- Staff can manually override it.
- Easy to display.
- The AI can assign priority during reconciliation.

#### Cons

- Priority can become stale.
  - Example: a “pending checkout safe issue” is critical before 8am, but after the guest leaves it may no longer mean the same thing.
- If business rules improve later, old priorities do not automatically update.
- AI-generated priority may be inconsistent.
- Staff may trust the label too much even when the underlying context changed.

### Option B: Computed priority

Priority is calculated from:

- `status`
- `type`
- room/guest context
- age of issue
- keywords in description
- due/urgency hints
- unresolved carry-over count
- compliance/finance/safety risk

Example rough rules:

```text
critical:
  - safety/security issue
  - guest cannot leave/check out
  - passport/cash locked in safe
  - active leak near guest room
  - compliance deadline near/overdue

high:
  - unresolved guest-impacting maintenance
  - unresolved deposit/payment issue before checkout
  - disputed charge
  - damage charge without approval

medium:
  - pending investigation
  - normal follow-up
  - unresolved but not immediately guest-blocking

low:
  - resolved items
  - FYI notes
  - low-impact facilities notes
```

#### Pros

- More reliable for carry-over tasks.
- Priority changes automatically as time passes.
- Easier to refine rules without rewriting all rows.
- Better for “handover dashboard” because staff care about “what needs action now”.

#### Cons

- More implementation logic.
- Staff cannot directly set exact priority unless we add override.
- Some priorities need human judgment.

### Option C: Hybrid: computed priority + optional manual override

This is my best suggestion.

Store the facts, compute the default priority, and allow override later.

Database fields:

```ts
computed_priority: not stored, derived by API
priority_override: "critical" | "high" | "medium" | "low" | null
priority_reason: string generated by API
```

API response:

```json
{
  "id": "evt_...",
  "status": "unresolved",
  "type": "facilities",
  "room": null,
  "description": "Water leak in 2nd floor corridor near room 215...",
  "priority": "critical",
  "prioritySource": "computed",
  "priorityReason": "Active water leak near guest room and unresolved overnight."
}
```

If staff overrides:

```json
{
  "priority": "high",
  "prioritySource": "manual",
  "priorityReason": "Manually lowered after bucket/sign placed and building management contacted."
}
```

### Why hybrid is best here

For this hotel handover POC, priority is not just a static label. It depends on context and time.

For example:

- Room 208 safe issue should be **critical** because passport/cash are locked and the guest has a morning flight.
- Corridor leak near 215 should be **critical/high** while actively dripping, but lower once building management attends.
- Room 112 aircon is unresolved but probably **medium/high**, because the guest has already moved to 115.
- Room 309 deposit issue becomes **high** near checkout.
- Room 214 guest message with prompt injection should probably be **medium/pending**, not critical, despite alarming text inside the note.
- Resolved no-show charge becomes less urgent, but later dispute makes it **pending/high finance follow-up**.

So I recommend:

1. Compute priority dynamically in the backend.
2. Return `priorityReason` for transparency.
3. Store optional `priority_override`.
4. Do not ask AI to be the final authority on priority.
5. AI may suggest urgency, but backend deterministic rules should decide display priority.

This matches your AGENTS.md principle: choose practical solution, but keep the workflow traceable and debuggable.
